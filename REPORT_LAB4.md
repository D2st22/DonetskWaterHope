# Лабораторна робота 4 - Масштабування бекенда

## Опис стратегії масштабування системи

Для системи MakiivskyRodnichokForever обрано горизонтальне масштабування backend-сервера. Це означає, що запускається декілька однакових копій ASP.NET Core Web API, які виконують одну й ту саму функціональність для різних запитів користувачів. Перед копіями сервера розміщується load balancer на базі Nginx, який приймає HTTP-запити та розподіляє їх між доступними інстансами API.

Обрана стратегія підходить для системи контролю водоспоживання, оскільки backend не зберігає стан користувацької сесії у пам’яті конкретного сервера. Авторизація виконується через JWT-токен, а основні дані зберігаються централізовано в PostgreSQL. Завдяки цьому будь-який запит користувача може бути оброблений будь-якою копією backend-сервера.

Загальна схема масштабування:

```text
Web / Mobile / IoT
        |
        v
   Nginx Load Balancer
        |
        +--> API instance 1
        +--> API instance 2
        +--> API instance 3
        |
        v
    PostgreSQL Database
```

## Технічні рішення, які роблять масштабування можливим

Backend реалізовано як ASP.NET Core Web API та контейнеризовано за допомогою Docker. Для демонстрації масштабування підготовлено Docker Compose-конфігурацію, у якій можна змінювати кількість копій сервера командою `--scale api=N`.

Основні технічні рішення:

- Docker-контейнер для backend API;
- Nginx як load balancer перед кількома копіями API;
- PostgreSQL як спільне централізоване сховище даних;
- JWT-авторизація без серверної сесії;
- REST API для взаємодії Web-клієнта, мобільного застосунку та IoT-пристрою з backend;
- endpoint `/health` для перевірки доступності інстансів;
- заголовок `X-Backend-Instance`, який дозволяє побачити, яка копія API обробила конкретний запит;
- окремий режим `--migrate-only` для виконання міграцій бази даних перед запуском масштабованих API-інстансів.

Файли конфігурації:

```text
scaling/docker-compose.scale.yml
scaling/nginx.conf
scaling/locust/locustfile.py
```

## Аналіз вузьких місць

Першим вузьким місцем при збільшенні навантаження може стати база даних PostgreSQL, оскільки всі копії backend-сервера звертаються до одного спільного сховища. Особливо це помітно для операцій читання історії споживання, списку лічильників та подій, а також для операцій запису показників від IoT-пристроїв.

Другим вузьким місцем є пул підключень до бази даних. Коли кількість API-інстансів збільшується, загальна кількість одночасних підключень до PostgreSQL також зростає. Якщо пул або база даних не налаштовані на таке навантаження, час відповіді починає збільшуватися.

Третім вузьким місцем можуть бути endpoint-и, які виконують складні запити з сортуванням, фільтрацією або приєднанням пов’язаних таблиць. Для таких запитів потрібні індекси в базі даних і оптимізація SQL-запитів.

Nginx load balancer у цій схемі не є основним вузьким місцем для лабораторного навантаження, оскільки він виконує лише розподіл HTTP-запитів між backend-інстансами.

## Опис навантажувального тестування

Для навантажувального тестування використовується Locust. Тест імітує користувачів, які авторизуються в системі та виконують типові запити:

- перевірка `/health`;
- отримання списку лічильників;
- отримання історії споживання;
- отримання попереджень;
- отримання звернень до підтримки.

Приклад запуску системи з одним API-інстансом:

```bash
docker compose -f scaling/docker-compose.scale.yml up --build --scale api=1
```

Приклад запуску з трьома API-інстансами:

```bash
docker compose -f scaling/docker-compose.scale.yml up --build --scale api=3
```

Locust запускається командою:

```bash
docker compose -f scaling/docker-compose.scale.yml --profile loadtest up --build --scale api=3
```

Після запуску Locust Web UI доступний за адресою:

```text
http://localhost:8089
```

Очікуваний результат полягає в тому, що зі збільшенням кількості backend-інстансів система здатна обробляти більшу кількість запитів за секунду. При цьому база даних залишається спільною, тому після певного рівня навантаження приріст продуктивності буде обмежений саме можливостями PostgreSQL та кількістю доступних підключень.

## Висновок

У лабораторній роботі було показано горизонтальне масштабування backend-частини системи MakiivskyRodnichokForever. Backend може запускатися у декількох однакових копіях, а Nginx розподіляє запити між ними. Завдяки JWT-авторизації та централізованому збереженню даних у PostgreSQL окремі інстанси API залишаються взаємозамінними.

Підготовлена конфігурація дозволяє змінювати кількість серверів, перевіряти розподіл запитів через `X-Backend-Instance` та проводити навантажувальне тестування за допомогою Locust.
